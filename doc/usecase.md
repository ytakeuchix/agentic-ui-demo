# 2026年版 AI経費精算システム：業務ユースケースシナリオ

本ドキュメントは、経費精算エージェントによる自律的な業務遂行（Zero UI）を軸に、文脈に応じて動的な入力フォーム（GenUI）や人間との共創空間（ホストアプリケーション：Canvas UI）を生成・提供する「エージェンティックUI」モデルに基づいた業務プロセスを定義する。

## 2026年AIエージェントUI/UX戦略の動向

2026年のUI/UXは、固定された「アプリの画面」から、経費精算エージェントが文脈に応じて生成・配信する「使い捨ての道具（GenUI）」へと大きくシフトしつつある。

- **Zero UI の普及**: 定型業務はバックグラウンドで自律実行され、インターフェースそのものが消失する。
- **空間的共創（Canvas）**: 複雑な意思決定においてのみ、経費精算エージェントと人間が思考を同期させる共有空間（Canvas）が起動する。
- **透明性の担保**: 経費精算エージェントの判断根拠（推論トレース）を可視化し、ブラックボックスを解消することで高い信頼性を構築する。

## 前提条件：ホストアプリケーションとUIの役割定義

本シナリオでは、経費精算エージェントが以下の役割分担でユーザーインターフェースを配信・制御することを前提とする。

- **ホストアプリケーション（実行基盤）**:
  - ユーザーが日常的に滞在するプラットフォーム（Slack, Google Workspace, 企業ポータル等）。
  - エージェントからの通知を受け取り、軽量なUI部品を直接描画する「器」として機能する。
- **GenUI（動的生成UI）**:
  - ホストアプリケーションのタイムラインやチャット内に、エージェントが動的に差し込む簡易UI部品（入力ボタン、セレクトボックス等）。
  - 特定の項目補完など、数秒で終わる「使い捨ての作業」に最適化されている。
- **Canvas UI（共有作業空間）**:
  - ホストアプリケーションから呼び出される、より高度でリッチな作業空間。
  - 大量のデータ比較や承認判断など、ホスト側の簡易UIでは認知負荷が高い場合に起動する。

## 設計コンセプト

2026年のAIエージェント戦略に基づき、実務上の「例外」や「責任」に耐えうる以下の原則を採用している。

- **人間の集中力の保護（Attention Protection）**:
  - 決済のたびに通知を飛ばさず一日の終わりにサマリー提示する。さらに、案件が多数存在する状況においても、経費精算エージェントが適切に情報の密度を制御する。
- **責任共有モデル（Shared Responsibility）**:
  - 経費精算エージェントは「高精度な推薦者」であり、最終的な法的・組織的責任は申請者（真実性の宣誓）と承認者（意思決定）にあることを操作画面上の確認メッセージ（案内文）で明示し、ガバナンスを維持する。
- **プライバシー・バイ・デザイン**:
  - 経費精算エージェントのデータアクセスは、ユーザーが許可した業務時間・キーワードに基づき制限される（オプトイン方式）。
- **法的真実性とレジリエンス**:
  - インボイス判定等の自動化に加え、規程改定時の「ポリシースナップショット参照」や、システム障害時の「一時的な送信待ち処理と状況報告」により、外部環境の変化や不具合に左右されない業務継続性を担保する。

## シナリオ1：【申請編】AIによる自律的な精算ドラフト作成

### 概要
経費精算エージェントがMCPを介して複数のSaaSを統合し、規程の変わり目やシステム障害も考慮してバックグラウンド入力を完結させる。

### メインフロー
1. 経費精算エージェントが、MCP（Model Context Protocol）経由でConcurへの法人カード利用通知をリアルタイム検知する。
2. 経費精算エージェントが領収書保存用の共有フォルダ内などを探索し、該当する領収書画像を特定する。画像内の日付に基づき、当該時点のポリシースナップショット（有効だった規程）を自動選択し、インボイス判定および規程照合を行う。
3. 経費精算エージェントが、過去の精算DBとクロスチェックを行い「二重申請」を自動検証する。
4. 経費精算エージェントが、カレンダー、フライト追跡、店舗予約APIからMCP経由で補足エビデンスを取得し、決済データと突き合わせる。
5. 経費精算エージェントが、Concurへデータを下書きとして保存する。
6. 申請者は一日の終わりにサマリーを確認し、一括で「申請」を実行する。

### 例外フロー
- **緊急申請の割り込み処理**: 支払期限が直近である場合や、出張中などで即時の申請完了が必要な場合、申請者はホストアプリケーション（Slack等）からエージェントに対し「緊急申請」を指示できる。エージェントは即座にドラフト作成と規程照合を優先実行し、サマリーを即時提示する。
- **システム障害・通信エラー**: SaaS（Concur等）との通信不能時は、一時的にエージェント側の一時待機領域に保存し、ユーザーには「復旧後に自動送信します」という状況説明メッセージを表示する。
- **非同期エスカレーション判断**: 以下の場合、経費精算エージェントが内部でフラグを立て、Step 6のサマリー提示時にまとめて対応を依頼する。
  - **領収書の不足・不鮮明**: クラウド上等で証跡が見つからない、または画像が判読不能な場合。
  - **規程抵執**: 金額上限超過、利用制限への抵触。
  - **低確信度**: 推論確信度が閾値（例：85%）未満の場合。
  - **情報の不整合**: 決済データと領収書の内容に差異がある場合。

### 完了状態
申請者によるサマリー確認と実行を経て、システム（Concur）上で法的要件をクリアした精算データの「申請」が正式に完了している状態。

## シナリオ2：【申請編】サマリー確認時の動的フォームによる補完

### 概要
要確認案件が発生している際に、経費精算エージェントが対話的に情報を補完させ、入力を完結させる。

- **シナリオ名**: 重要度に基づく自動整理と補完
- **アクター**: 経費精算エージェント、申請者、ホストアプリケーション

### メインフロー
1. 経費精算エージェントが、ホストアプリケーションのチャット・通知インターフェース上にサマリーを提示し、申請者に確認を促す。
2. ユーザーが対象案件を選択すると、経費精算エージェントがその場に最適なGenUI（動的生成フォーム）を配信。例えば会食同席者不明時は、名刺管理データから推論した候補を選択肢として提示する。
3. 修正・入力完了時、UI上に「本内容は真実であることを宣誓します」という法的責任を再認させるボタンを表示。
4. 確定後、GenUIは即座に自動的に消去され、ユーザーはホストアプリケーションでの通常業務へ戻る。

### 例外フロー
- **大量の要確認案件が発生している場合（月末・決算期等）**: 経費精算エージェントが重要度に基づき自動整理したサマリーを提示し、優先順位をナビゲートする。
- **複雑な判断が必要な場合**: 簡易的なGenUIでの補完が困難な場合、エージェントはCanvas UI（専用の共有作業空間）を起動し、より詳細なデータ照合を促す。

### 完了状態
経費精算エージェントが配信したGenUIを通じて不足情報が補完され、内容が真実であることの確認・承諾を行い、システム（Concur）上での「申請」が正式に完了している状態。

## シナリオ3：【承認編】エビデンスに基づくレビューと承認

### 概要
承認者不在時や組織変更にも柔軟に対応し、法的責任を伴う判断を迅速に完了させる。

- **シナリオ名**: 状況に応じた承認者の自動設定と専用画面による一括承認
- **アクター**: 承認者、経費精算エージェント、ホストアプリケーション

### メインフロー
1. 経費精算エージェントが、承認者に対し、ホストアプリケーション経由で週次の精算サマリー通知を配信する。
2. 承認者が通知内のリンクを選択すると、ホストアプリケーションから遷移して、対象期間の全案件を俯瞰できる「Canvas UI（承認用共有キャンバス）」が起動する。
3. Canvas UIが、インボイス判定、規程照合、外部データ（搭乗ログ等）との照合結果を一画面に集約表示。
4. 承認者が「警告」案件を無視して承認しようとした場合、システムは「経費精算エージェントの警告を上書きする理由」の入力を強制し（安全性を確保するためのあえての確認手順）、監査ログに記録する。
5. 経費精算エージェントが承認処理を確定させ、銀行APIとの連携状況（振込予約日等）を処理状況の報告として表示する。

### 例外フロー
- **承認者不在時の代理設定**: 本来の承認者が不在（休暇等）の場合、MCPが権限マトリクスから「次の適切な代理承認者」を特定し、Canvasへの期間限定アクセスを自動付与する。

### 完了状態
承認者（または正当な代理承認者）が、Canvas UI上で提示された多角的な証拠に基づき、ガバナンスを遵守した上で承認判断を確定させ、支払い処理（銀行連携等）が正式に開始されている状態。
